# 分支管理与发布流程

## 分支说明

| 分支      | 用途                             | 谁可以操作 | CI 流水线       |
| --------- | -------------------------------- | ---------- | --------------- |
| `dev`     | 开发集成分支，合并日常开发代码   | 所有开发者 | 单元测试        |
| `test`    | 测试分支，部署测试环境供 QA 验证 | 所有开发者 | 单元测试 + 构建 |
| `release` | 发布分支，合并稳定代码后构建发布 | 发版负责人 | 单元测试 + 构建 |
| `master`  | 生产快照，始终与线上版本一致     | 发版负责人 | 不触发 CI       |

### 分支关系

```
dev（日常开发）→ test（测试验证）→ release（发布构建）→ master（生产快照 + tag）
```

- `dev` 是**唯一的日常开发目标分支**，所有功能代码都合并到这里
- `test` 是**测试环境分支**，合并 `dev` 的代码后自动构建，构建产物可从 GitLab 下载手动部署
- `release` 是**发布关卡**，测试通过后从 `test`（或 `dev`）合并到这里
- `master` 是**生产环境快照**，只接受来自 `release` 的合并，通过 tag 标记版本

## 日常开发流程

### 步骤

1. 在本地基于 `dev` 分支创建功能分支（可选，也可直接在 `dev` 上开发）
2. 完成开发后，提交代码并推送
3. 在 GitLab 上创建 MR 合并到 `dev`（或直接 push 到 `dev`）
4. CI 自动运行单元测试，通过后合并

### 提交规范

提交信息建议使用以下前缀：

| 前缀     | 用途     | 示例                          |
| -------- | -------- | ----------------------------- |
| feat     | 新功能   | `feat: 新增客户分析报表`      |
| fix      | 修复 Bug | `fix: 修复金额格式化精度问题` |
| chore    | 杂务     | `chore: 更新依赖版本`         |
| style    | 样式调整 | `style: 调整首页卡片间距`     |
| refactor | 重构     | `refactor: 重构权限校验逻辑`  |
| docs     | 文档     | `docs: 更新 API 调用说明`     |

### 本地质量保障

项目配置了 `husky` + `lint-staged`，每次 `git commit` 时自动执行：

- **ESLint**：检查并自动修复 `.vue / .js / .ts` 文件
- **Stylelint**：检查并自动修复 `.css / .scss` 文件
- **Prettier**：格式化 `.json / .md` 文件
- **文件名检查**：确保文件命名符合规范

如果检查不通过，commit 会被阻止，需要修复后重新提交。

## 测试验证流程

### 操作步骤

1. 功能在 `dev` 分支开发完成后，创建 MR：`dev → test`
2. 合并后 CI 自动执行：**单元测试 → 构建 H5**
3. 从 CI 的 `build_h5` Job 中下载构建产物（`dist/build/h5/`），手动部署到测试环境
4. 测试环境地址：http://10.18.14.225:8080/wxdist/index.html

## 发版流程

### 前提条件

- `test` 分支上的功能已在测试环境验证通过
- 确认本次要发布的功能范围

### 操作步骤

#### 第一步：dev → release（GitLab 网页操作）

1. 打开 GitLab 项目页面
2. 点击 **Merge Requests → New merge request**
3. Source branch 选择 `dev`，Target branch 选择 `release`
4. 填写标题，如：`发版：v1.2.0 新增XX功能`
5. 点击 **Create merge request → Merge**
6. CI 自动执行：**单元测试 → 构建 H5**
7. 等待 CI 通过，构建产物自动保存在 Artifacts 中

#### 第二步：部署生产环境

1. 从 CI 的 `build_h5` Job 中下载构建产物（`dist/build/h5/`）
2. 部署到生产环境服务器
3. 验证线上功能正常

#### 第三步：release → master（GitLab 网页操作）

1. 确认生产环境部署成功
2. 在 GitLab 创建 MR：Source 选择 `release`，Target 选择 `master`
3. 合并 MR

#### 第四步：打 Tag（GitLab 网页操作）

1. 在 GitLab 进入 **Repository → Tags → New Tag**
2. 填写信息：
   - **Tag name**：`v1.2.0`（语义化版本号）
   - **Create from**：`master`
   - **Message**：简要描述本次发布内容
3. 点击 **Create tag**

## Tag 版本号规范

采用 [语义化版本（SemVer）](https://semver.org/lang/zh-CN/) 格式：

```
v主版本号.次版本号.修订号
```

| 位置       | 何时递增                   | 示例                |
| ---------- | -------------------------- | ------------------- |
| **主版本** | 大改版、不兼容的架构变更   | v1.0.0 → v**2**.0.0 |
| **次版本** | 新增功能、模块，向下兼容   | v1.0.0 → v1.**1**.0 |
| **修订号** | Bug 修复、小调整、文案修改 | v1.0.0 → v1.0.**1** |

### 命名示例

```
v1.0.0  - 首次正式发布
v1.1.0  - 新增客户分析报表功能
v1.1.1  - 修复报表金额显示精度问题
v1.2.0  - 新增消息推送、优化首页加载
v2.0.0  - 架构升级（如从 Vue 2 迁移到 Vue 3）
```

## CI 流水线说明

### dev 分支

```
unit_test
```

- **触发条件**：推送到 `dev` 或 MR 目标为 `dev`
- **单元测试失败会阻塞合并**

### test 分支

```
unit_test → build_h5
```

- **触发条件**：推送到 `test` 或 MR 目标为 `test`
- **单元测试失败会阻塞构建**
- **构建产物保留 3 天**，可从 GitLab Job 页面下载后手动部署到测试环境
- ⚠️ 自动部署待 Runner 网络就绪后恢复

### release 分支

```
unit_test → build_h5
```

- **触发条件**：推送到 `release` 或 MR 目标为 `release`
- **单元测试失败会阻塞构建**
- **构建产物保留 3 天**，可从 GitLab Job 页面下载

### master 分支

- **不触发任何 CI**，仅作为生产快照和 tag 载体

## 紧急修复（Hotfix）

当生产环境出现紧急 Bug 需要快速修复时：

1. 基于 `master` 创建修复分支（或直接在 `dev` 上修复）
2. 修复代码合并到 `dev`
3. 按照正常发版流程走 `dev → release → master`
4. 修订号 +1（如 v1.2.0 → v1.2.1）

> 即使是紧急修复，也建议走完整流程，确保 CI 验证通过后再发布。

## 常见问题

### Q: 可以直接 push 到 dev 还是必须走 MR？

现阶段可以直接 push 到 `dev`。如果团队规模扩大，建议切换到 feature 分支 + MR 的模式，开启 Code Review。

### Q: release 分支 CI 失败了怎么办？

回到 `dev` 分支修复问题，修复后重新走 `dev → release` 的 MR 流程。**不要直接在 release 分支上修代码。**

### Q: 忘记打 tag 了怎么办？

随时可以补打。在 GitLab 的 **Repository → Tags → New Tag** 页面，选择 `master` 分支上对应的 commit 即可。

### Q: dev 和 release 代码不同步怎么办？

正常情况下 release 的代码是 dev 的子集。如果出现分叉，以 `dev` 为准，重新创建 `dev → release` 的 MR 覆盖即可。
